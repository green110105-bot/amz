## 1. 模块概述

### 1.1 价值主张

> **异常、差评、竞品都不漏掉，并且每个事件都自带"该怎么办"。预警 → 响应 → 闭环。**

### 1.2 核心特性

| 特性 | 描述 |
|---|---|
| 三位一体 | 自家异常 + Review + 竞品 一站式 |
| 全维监控 | **22 类**自家异常（扩展） + Review 实时分析 + 竞品 9 维监控（扩展） |
| 严重度分层 | P0 紧急 / P1 重要 / P2 关注 |
| 推送多通道 | App + 邮件（Phase 1）→ 微信 / 企微（Phase 2）|
| 处置内嵌 | 每个事件附"该怎么办"，可一键处置 |
| 申诉自动化 | 恶意评论申诉文案 + 客户挽回邮件草稿 |
| 闭环跟踪 | 处置后跟踪效果 |
| 安静模式 | 静默时段 + 严重度分级路由 |
| **责任人分配** | **异常自动派单到运营 / 采购 / 财务，含 SLA 倒计时** |
| **关联性分析** | **多个异常 AI 关联（同根因聚合）** |
| **图像变化检测** | **Listing 图像被改的多模态视觉对比** |
| **跟卖检测** | **跟卖出现/消失的实时监测 + 申诉支持** |
| **侵权预警** | **品牌 / 专利 / 版权侵权告警 + 应对** |
| **知识沉淀** | **处置成功案例自动入知识库** |
| **Vine 跟踪** | **Vine 评论单独标记 + 趋势** |
| **复盘归档** | **重大事件自动生成事后复盘报告** |
| **库容监控** | **IPI / 库容利用率 / 触线预警** |
| **B2B 异常** | **企业买家大额订单的特殊监控** |

### 1.3 不做的事

- ❌ 不做客户邮件管理（领星已成熟）
- ❌ 不做退款 / Case 处理流程（用 Amazon 后台）
- ❌ 申诉 / 挽回邮件仅生成草稿，不自动发送（除非全自动模式）
- ❌ 不做实时聊天客服

```

    • 并入数据底座，被其他模块共享
```

### 6.1a 自适应阈值与误报控制（关键质量保障）

> ⚠️ 22 类异常的"误报率 ≤ 10%"目标过于严格，单类实际可能 20-30%。**必须有自适应机制**避免用户麻木。

#### 自适应阈值

每个异常规则有"动态阈值"：
- 初始用类目均值或全平台均值
- 每周根据该 SKU / 该租户的历史数据自动调整 σ
- 用户标"误报"3 次 → 该规则该范围阈值放宽 25%
- 用户连续 5 次"立即处理"+ 标"有效" → 阈值收紧 10%

#### 反馈降权机制

```yaml
异常通知伴随两个按钮：
  [✓ 有效，已处理]   → AI 模型加权"此场景算异常"
  [✗ 误报]           → 1 次降权 25%；3 次该规则该 SKU 静音 30 天

每月生成"异常质量报告"：
  - 各类异常的"采纳率"（用户标有效 / 总通知数）
  - 自动调阈值的变化记录
  - 提示用户"以下规则误报率高，建议手动调整"
```

#### 严重度路由策略

避免"22 类全 P1 一次性轰炸"：
- P0 阈值严格，确保只有真正紧急的事件触发
- P1 默认聚合（每 15 分钟一次摘要，而不是即时）
- P2 默认每日摘要

### 6.4 各异常类型的详细处置

#### 6.4.1 A4 Buy Box 丢失（P0）

**检测：**
- 数据源：SP-API ItemOffersBatch（每 10 分钟轮询）
- 规则：buybox_winner != self_seller_id 持续 ≥ 30 min

**处置建议：**
- 主因分析（价格 / 库存周转 / 跟卖）
- A. 跟价至 BB 价 → 自动 / 手动可选
- B. 启动 Coupon
- C. 静观（如 BB 价过低不值得）

**跟踪：**
- 处置后每 5 分钟检查 BB 状态
- 30 分钟内拿回 → 'resolved'
- 未拿回 → 升级追踪

#### 6.4.2 A5 Listing 被改（P0）

**检测：**
- 数据源：SP-API CatalogItems（每小时拉取）
- 规则：title / bullets / images / a+ 内容 hash 变化

**处置建议：**
- 显示变更对比（旧 vs 新）
- A. 提交 Case 申诉（系统起草）
- B. 立即恢复（如有 Brand Registry，可直接覆盖）

**跟踪：**
- 提交 Case 后跟踪 Case 状态
- 恢复后跟踪是否再次被改

#### 6.4.3 A1 销量异常（P1）

**检测：**
- 数据源：order_profits 或 inventory snapshots
- 规则：单 SKU 24h 销量 |Z-score| > 3

**复核：**
- AI 检查是否为合理事件（如本 SKU 启动促销 / 节日 / 活动）
- 排除已知事件后才报警

**处置建议：**
- 根因分析（广告变化 / 价格变化 / 排名变化 / 政策事件）
- 关联到具体根因模块（M3 / M2 / M1）

#### 6.4.4 A8 退款激增（P1）

**检测：**
- 单 SKU 24h 退款率 > 10% 且退款数 ≥ 3

**处置建议：**
- 触发 Review 分析（看是否对应 1-3 星新评论）
- 触发 M1 Listing 检查（是否有误导）
- 触发供应链检查

#### 6.4.5 A11 账户健康（P0）

**检测：**
- 数据源：SP-API AccountHealth API
- 规则：ODR > 1% / Late Shipment Rate > 4% / Cancellation > 2.5% / VTR (有效追踪) < 95%

**处置建议：**
- 立即定位问题订单
- 起草 POA（Plan of Action）申诉文案
- 提示用户立即处理

#### 6.4.6 A15 跟卖出现（P0）

**检测：**
- 每小时检查每个自家 ASIN 的 Offers 列表
- 出现非自家 seller_id → 触发

**处置建议：**

```
🔴 跟卖警报：B0XXX
跟卖者: SellerXYZ ｜ 价格 $19.99（你 $24.99）
─────────────────────────────────
AI 分析：
  跟卖类型: 价格竞争（非假货）
  风险: 抢 BB → 销量损失约 ¥X / 小时

处置选项：
A. Test Buy（测试购买，验证是否假货）
   [一键 Test Buy] (约 7-14 天到货)
   
B. 直接申诉（如有 Brand Registry）
   [起草侵权申诉]
   
C. 价格防御（跟到 $19.99 抢回 BB）
   [跳到 M2 跟价测算]
   
D. 综合处置（推荐）
   1. 立即 Test Buy
   2. 同步跟价至 $20.99（暂时止血）
   3. 收到货确认后申诉
   [一键执行综合]
```

**Test Buy 流程（指导模式，非系统自动下单）：**

> ⚠️ **本系统不直接代用户下单**。原因：涉及买家账户授权 + 资金 + 发货地址 + 法律责任。改为指导用户自己完成 + 系统跟踪流程。

```
Step 1: 系统准备（自动）
  - 生成 Test Buy 指引（哪个 ASIN / 哪个 seller offer / 推荐数量 = 1）
  - 生成 ASIN 详情页直达链接
  - 提示用户用"非主账号 + 非品牌邮箱"下单（避免被对方识别）
  
Step 2: 用户操作（手动）
  - 用户在 Amazon 上自行下单
  - 在系统填入 Order ID + Tracking Number
  
Step 3: 系统跟踪（自动）
  - 跟踪物流到货时间
  - 到货前推送提醒"准备好开箱拍照对比"
  
Step 4: 验货（用户）
  - 用户上传开箱照片 / 视频
  - AI 多模态对比：与你的真品差异点
  - 用户最终判定：✓ 假货 / ⚠ 描述不符 / ✗ 真品（仅价格竞争）
  
Step 5: 申诉路径（半自动）
  - 假货 → 自动起草 A-to-z 投诉 + IP 侵权举报草稿，用户审后自行提交
  - 描述不符 → 起草违反 Listing 政策的申诉草稿
  - 真品 → 跳转 M2 跟价测算（仅价格防御）
```

**强调（UI 必须显示）：**
> ⚠️ Test Buy 由你自主完成，本产品仅提供流程指导和文案起草。下单成本（货款 + 运费）由你承担。后续退款由你与亚马逊客服处理。本产品**不代为操作买家账户**。

#### 6.4.7 A16 侵权告警（P0）

**来源：**
- Amazon Brand Registry 内的"Report Violation"
- 第三方监控（IP Accelerator）
- 手动用户上报

**处置建议：**
- 自动归类（商标 / 专利 / 版权 / 假货）
- 自动起草 IP 投诉
- 提示是否需法律咨询

##### 未注册 Brand Registry 的用户路径

> ⚠️ Brand Registry 是亚马逊侵权处置的核心工具。**未注册的卖家受限于此功能**。

- **检测仍可用**：来自第三方 IP 监控 + 用户手动上报
- **自动起草仍可用**：但举报渠道受限（仅可走 Amazon Seller Support 而非 Brand Registry 加速通道）
- **UI 提示**：检测到侵权时，未注册用户会看到"立即注册 Brand Registry 加速处置 [指引]"
- **生成的申诉文案**会针对"未注册"路径优化（用 Seller Support 通用模板而非 Brand Registry 专属）

> **法律免责**（必须在 UI 显示）：本系统起草的申诉文案为参考稿，不构成法律意见。涉及商标/专利的复杂案件建议咨询专业律师。

#### 6.4.8 A18 退款类型异常（P1）

  - "描述不符"
  - "买错 / 不需要"
- 单 SKU 单类型 30 天比例突增

**处置：**
- "破损"上升 → 检查包装（可能联动供应链）
- "描述不符"上升 → 推送 M1 优化 Listing
- "物流问题"上升 → 检查 FBA 仓发货延迟
- "不喜欢"上升 → 类目正常波动，关注趋势

**模板回复（默认草稿模式）：**

> ⚠️ **错误的客服回复直接影响 ODR 和账户健康**。本系统**默认仅生成草稿**，不自动发送。

**三档主权（用户配置）：**

| 档位 | 行为 | 适用 |
|---|---|---|
| **草稿模式（默认）** | AI 起草回复，用户必须审 + 手动点"发送" | 全部场景 |
| **预批准模板** | 用户预先审过 N 个模板（如"物流查询"、"退货流程"），系统识别消息类型后自动用对应模板回复 | 高频简单问询 |
| **全自动**（仅 P2 增值） | 系统全程自动；用户事后审计 | 不推荐除非超大店铺 |

**预批准模板的护栏：**
- 模板必须经过用户至少 1 次"审核 + 批准"才能进入自动池
- 模板覆盖范围严格匹配（消息分类置信度 > 0.85 才用模板）
- 涉及"退款 / 投诉 / 法律 / 索赔"关键词 → 强制回到草稿模式
- 用户可随时撤销某模板的自动状态
- 每月生成"自动回复审计报告"，列出全部自动发送内容供回顾

**审计：**
- 自动发送的每条回复必须记录到 `auto_execution_logs`
- 24h 内可一键撤回（亚马逊 Buyer-Seller Messaging 支持发送后编辑/删除）
- 错误回复达成 "买家投诉" 等 → 立即冻结该模板

## 14. 业务规则

### 14.1 异常去重

- 同 type + scope 24h 内只触发一次
- 已存在 'pending'/'acknowledged' 异常时，新触发更新 evidence + 严重度

### 14.2 严重度升级

- P1 累计 24h 未处理 → 升级 P0
- P2 累计 7 天未处理 → 升级 P1

### 14.3 自动处置

| 异常类型 | 全自动允许 | 半自动 | 全手动 |
|---|---|---|---|
| A4 BB 丢失（小幅跟价） | ✓ | ✓ | ✓ |
| A4 BB 丢失（大幅跟价 > 10%） | ✗ | ✓ | ✓ |
| A11 账户健康 POA | ✗ | ✗（仅起草） | ✓ |
| 其他 | 按类型默认 | 按类型默认 | ✓ |

### 14.4 Review 申诉

- 仅对 1-3 星 Review 评估申诉
- AI 置信度 < 70% 不主动推荐
- 用户提交后跟踪 30 天 Amazon 响应

### 14.5 Review 挽回

- 仅对 1-3 星 Verified Purchase Review
- 24h 内仅发送 1 次挽回邮件
- 通过 Amazon Buyer-Seller Messaging（合规）

### 14.6 竞品监控

- 每 SKU 监控竞品数有套餐上限
- 第三方数据 API 配额按租户分配
- 配额耗尽提示用户升级

## 15. 边界条件与异常

### 15.1 数据延迟

| 数据 | 实际延迟 | 处理 |
|---|---|---|
| Buy Box（10 分钟轮询） | 最差 10 分钟 | 通知用户 SLA |
| Review（6 小时拉取） | 最差 6 小时 | 摘要通知 |
| 竞品（1-12 小时） | 最差 12 小时 | 通知用户 SLA |
| 政策公告 | 不可控 | 主动 RSS / 群组 |

### 15.2 第三方 API 故障

| 故障 | 处理 |
|---|---|
| Keepa 中断 | 用 SellerSprite 兜底 + 提示 |
| SellerSprite 中断 | 用 Helium 10 兜底 |
| 双中断 | 缓存可用数据，提示监控降级 |

### 15.3 通知通道故障

| 故障 | 处理 |
|---|---|
| 邮件送达失败 | 重试 3 次 + fallback to 站内 |
| 微信推送失败（用户解绑等） | 通知用户重新绑定 |
| 高峰队列堆积 | 优先发送 P0，P2 延迟 |

### 15.4 用户操作异常

| 情况 | 处理 |
|---|---|
| 异常处置时多人同操作 | 乐观锁 + 提示 |
| 自动处置后用户改回 | 标记"用户撤销"，停止该类自动 |
| Review 申诉重复提交 | 第二次拒绝 + 提示 |
