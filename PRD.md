## 1. 产品概述

### 1.1 一句话定位

> **"给亚马逊卖家的 AI 操盘手 —— 一个人 + AI = 一个运营团队。"**

> ⚠️ **关键说明（必须在所有产品文案、UI、首页明确）：**
>
> "AI 操盘手"是产品**对外品牌叙事**。在用户体验和法律层面：
> - AI 给**判断 + 给依据 + 给可执行动作**
> - **决定权和最终操作权始终在用户**（除非用户主动开启"全自动"模式）
> - 全自动模式有明确**护栏**（金额/类型/SKU 白名单）+ **审计回溯** + **一键回滚**
> - AI 建议错了不构成法律责任，所有 UI 弹窗、文档、合同需有此免责说明
>
> 内部更准确的定位是 **"AI 副驾驶 + 可选自动驾驶"**。对外营销可继续用"操盘手"，但功能交付必须遵守此原则。

### 1.2 解决的核心问题

亚马逊卖家（尤其是中小卖家）的真实困境：

1. **信息过载、决策无力**：SP-API、Ads API、Review、竞品数据散落在 5+ 工具中，但**没有人替你下判断**。
2. **看似赚了其实在亏**：销售、广告、退货、仓储费、汇率分散记账，单 SKU/单订单/单关键词的真实利润几乎没人算得清。
3. **运营靠经验**：好运营年薪 ≥30 万，中小卖家请不起；新手卖家凭直觉操作，亏损率高。
4. **执行层 ERP 已饱和**：领星、店小秘、马帮等已经把"流程管理"做透，但"决策建议"是空白。

### 1.3 我们的差异化

| 维度 | 现有 ERP（领星等） | 本产品 |
|---|---|---|
| 定位 | 执行层（流程、记录、看板） | **决策层（判断、建议、自动化）** |
| 用户主张 | "管好你的店" | "替你做决策" |
| 核心能力 | 数据展示 | **AI 给判断 + 给理由 + 可执行** |
| 关键交付 | 报表 | **可解释的决策卡片** |
| 主权设计 | 用户自主 | **三档主权（手动 / 半自动 / 全自动）** |

### 1.4 北极星指标体系

#### 总产品北极星（双层）

| 层级 | 指标 | 测量方式 | 周期 |
|---|---|---|---|
| **价值层（终极）** | 月度 AI 贡献净利润 | 被采纳的 AI 决策关联 SKU 在 N 天内的利润增量（归因法）+ 季度 A/B 校准 | 月 / 季 |
| **行为层（领先）** | 周采纳决策数 / 活跃 SKU | 用户 7 日内点击"采纳/执行"的 AI 建议数 ÷ 活跃 SKU 数 | 周 |

#### 各模块北极星

| 模块 | 北极星 | 代理指标（短周期） |
|---|---|---|
| M1 Listing | 优化后 CVR lift（A/B 14d） | 评分提升幅度、采纳建议数 |
| M2 利润/库存 | 月度可挽回利润额、断货率+滞销率综合下降 | 漏点识别数 → 修复转化率、补货 MAPE |
| M3 广告 | 利润口径 ROAS 提升、ACOS 降幅（销售不降） | 周期识别准确率、操作采纳率 |
| M4 监控 | 异常响应时长↓、差评落地改进数、申诉成功率 | 推送→打开率、推送→操作率 |

### 1.5 业务价值假设

> 假设一个中卖家（30 SKU、月销 ¥80 万、毛利 25%），使用本产品后：
> - 利润可挽回点：3-5%（漏点修复）+ 2-4%（广告优化）+ 1-3%（库存周转）+ 1-2%（CVR 提升）
> - **合计可期净利润提升 7-14%**，即每月 1.4-3.2 万
> - 客户愿付价格：每月 999-1999（价值 / 价格 = 10x+），转化率乐观

## 2. 目标用户与场景

### 2.1 用户分层

| 分层 | SKU 数 | 月 GMV | 团队 | 核心痛点 | 适配套餐 |
|---|---|---|---|---|---|
| 个体小卖 | 1-10 | < ¥10 万 | 1 人 | 不懂运营、没钱请人 | 入门 |
| 成长中卖 | 10-100 | ¥10-200 万 | 1-5 人 | 数据多管不过来、决策慢 | 增长 |
| 规模大卖 | 100-1000+ | ¥200 万+ | 5+ 人、多店多国家 | 团队协作、批量、自定义、利润不透明 | 操盘 |

### 2.2 用户画像（典型 Persona）

#### Persona A：李欢欢（个体小卖，新手）
- 35 岁，刚从外贸转独立卖家
- 1 个 US 站、5 个 SKU、月销 5 万
- 痛点：不会调广告、看不懂报表、Listing 写得不专业
- 期望：有个"师傅"告诉她每天该做什么

#### Persona B：王浩（成长中卖，运营出身）
- 30 岁，前大卖运营，自己出来开店 3 年
- 2 个店铺、40 SKU、月销 80 万
- 痛点：广告 ACOS 不稳、补货老踩坑、利润算不清
- 期望：替他过滤噪音、聚焦真正需要决策的事

#### Persona C：赵总（大卖创始人）
- 42 岁，5 年亚马逊老兵
- 10 个店铺、4 个国家站、200+ SKU、年销 5000 万
- 团队：3 个运营、1 个采购、1 个财务
- 痛点：团队执行不一致、各项目利润看不清、竞品反应慢
- 期望：统一的决策中枢 + 团队协作 + 可定制规则

## 5. 模块 M1：AI Listing 迭代优化室

### 5.1 价值主张

> **不是一键生成，是"评分 → AI 改 → 评分 → 再改"的对话式表单迭代。运营和 AI 协作，看得见 AI 的判断逻辑。**

### 5.2 用户故事

- **US-M1-1** 作为运营，我希望知道我的 Listing 现在到底"差在哪"，而不是只看一个总分。
- **US-M1-2** 作为运营，我希望 AI 改完后我能选择采纳/拒绝/微调每一处改动。
- **US-M1-3** 作为运营，我希望每一轮迭代都有评分变化，能看到"是不是真的变好了"。
- **US-M1-4** 作为运营，我希望最后能一键把优化版同步到亚马逊后台 + 启动 A/B 测试。
- **US-M1-5** 作为运营，我希望能基于头部竞品和真实用户痛点（Top Review）来优化，而不是 AI 凭空想象。

### 5.3 评分体系（百分制 5 维度）

| 维度 | 权重 | 评估方式 |
|---|---|---|
| **关键词覆盖度** | 25% | SP-API 搜索词报告 → 高曝光/高转化词在 Listing 中的覆盖率 |
| **卖点清晰度** | 20% | LLM 评估五点是否突出独特卖点（USP）、是否有实证、是否回应购买决策因素 |
| **用户痛点对齐** | 20% | LLM 对比 Listing 表述 vs Top Review 真实关注点（差评中提到的、好评中提到的） |
| **视觉与 A+** | 15% | 多模态模型评估主图、A+ 视觉冲击、信息密度、CTA |
| **转化诱因** | 20% | 社会证明、紧迫感、对比优势、风险消除（退换、保修） |

每一维度有明细子项（如关键词覆盖度下的"主词"/"长尾词"/"购物意图词"），评分时附详细解释。

### 5.4 迭代流程（4 阶段）

```
[阶段 1：诊断]
输入：当前 ASIN
↓
拉取：当前 Listing + 头部竞品 3-5 个 + 搜索词报告 + Top Review（自家+竞品）
↓
AI 输出：5 维度评分 + 总分 + 「最关键的 3 个改进点」（按 ROI 排序）

[阶段 2：协商]
运营选择 1 个改进点 → AI 给 3 个改写方案（不同风格/侧重）
运营操作选项：
  - 直接采用 A/B/C
  - 在 A/B/C 上微调
  - 反馈"再来一版，我希望更突出 X"
  - 跳过这个改进点

[阶段 3：重评]
改写完成 → 重新打分
展示评分对比卡：哪个维度涨了/降了/没动
列出本轮所有改动 + 理由

[阶段 4：循环或交付]
继续选下一个改进点 → 回到阶段 2
or 完成 → 进入交付：
  - 一键同步到亚马逊（通过 SP-API ListingItems API）
  - 一键启动 A/B 测试（亚马逊 Manage Your Experiments，仅支持主图 / A+）
  - 14 天后自动回收 CVR 数据，给"是否提升"的结论
```

### 5.5 数据来源

| 数据 | 来源 | 频率 |
|---|---|---|
| 当前 Listing | SP-API ListingsItems / Catalog Items | 实时拉 |
| 竞品 Listing | Keepa / SellerSprite + 自动选定头部 ASIN | 天级缓存 |
| 搜索词报告 | SP-API Reports（Brand Analytics、Search Query Performance） | 周级 |
| Top Review | 第三方（SellerSprite Review API） | 天级缓存 |
| Listing 历史变更 | 自家系统记录 | 实时 |
| A/B 实验结果 | SP-API Experiments API | 实验结束后 |

### 5.6 AI 能力清单

| 能力 | 实现 | 模型 |
|---|---|---|
| 关键词覆盖率分析 | 规则 + Embedding 语义匹配 | text-embedding-3 |
| 五点重写 | LLM Few-shot prompt（按类目模板） | Claude Opus / GPT-4 |
| A+ 文案重写 | LLM | Claude Opus |
| 主图建议 | 多模态评估 | Claude Opus / GPT-4V |
| 关键词补充建议 | 搜索词报告 + 类目热词 | 规则 + LLM |
| 评分推理链 | LLM CoT，每条评分给依据 | Claude Opus |

## 6. 模块 M2：实时利润中枢 + 库存决策

### 6.1 价值主张

> **"看似赚了其实在亏" —— 把利润黑洞照亮。每一块钱赚在哪、亏在哪、压在哪，全部实时可见。同时把"补货 / 滞销 / 跟价"决策建立在真实利润而不是销售之上。**

### 6.2 真实痛点深挖

亚马逊卖家"看不清利润"的 8 个真实漏点：

| # | 漏点 | 影响 |
|---|---|---|
| 1 | 退货周期 30 天，今天的"利润"是谎言 | 实际净利润可能比账面低 5-15% |
| 2 | 长期仓储费每月 5 号、12 号悄悄扣 | 滞销 SKU 实际亏损 |
| 3 | 退货处理费、移除费、配送费分散 | 单 SKU 真实成本看不清 |
| 4 | 广告 ACOS 是销售口径不是利润口径 | 看似 ACOS 30% 可以，实际毛利只有 25%，在亏 |
| 5 | 汇率损益没人记 | 跨境结汇可能吞掉 1-3% |
| 6 | 库存资金占用成本 | 压货 30 天等于年化 12% 利息 |
| 7 | 退款手续费 / Refund Commission | 复杂规则，人工算不清 |
| 8 | FBA 重新装箱、贴标、合规标签 | 各种隐藏运营费 |

### 6.3 用户故事

- **US-M2-1** 作为卖家，我要看到**单订单**真实利润（扣完所有费用）。
- **US-M2-2** 作为卖家，我要看到**单 SKU**当日 / 7 日 / 30 日滚动利润 + 利润率趋势。
- **US-M2-3** 作为卖家，我要看到**单广告活动 / 关键词**的利润口径 ROAS（不是销售口径）。
- **US-M2-4** 作为卖家，我要 AI 主动告诉我"这里漏钱了，去做 X"。
- **US-M2-5** 作为卖家，我要看到每个 SKU 的**资金周转天数**：钱压在这上面多久回本？
- **US-M2-6** 作为卖家，我要 AI 给"补货 / 不补 / 清库 / 移除"的决策，依据是利润而不是销量。
- **US-M2-7** 作为卖家，我要在跟竞品价时知道"跟到什么价我还赚钱"。
- **US-M2-8** 作为卖家，我要导出到自己的财务系统对账。

### 6.4 利润核算公式（精确版）

```
单订单净利润 =
  + 售价 × 数量
  − Amazon 销售佣金
  − FBA 配送费
  − 退货预提（按 SKU 历史退货率）
  − 退货处理费（按 SKU 历史）
  − 仓储费分摊（按当月入仓体积/天数）
  − 长期仓储费（如适用）
  − 广告分摊（按 SKU 当日广告费 ÷ 当日订单数）
  − 采购成本（用户录入 / 估算）
  − 头程运费分摊（用户录入）
  − 退款手续费（如已退款）
  − 汇率损益（结汇时计算）
  − 增值税 / VAT（按站点）
  − 其他费用（移除费、贴标费等）

单 SKU 滚动利润 = ∑ 该 SKU 时段内所有订单净利润
```

### 6.5 关键功能模块

#### 6.5.1 利润仪表盘

| 视图 | 维度 | 数据展示 |
|---|---|---|
| 总览 | 时间（日/周/月）× 店铺 × 国家 | GMV、净利润、利润率、订单数、退款数、广告费、退货率 |
| SKU 级 | SKU × 时间 | 单 SKU 利润、利润率、销量、资金周转、滞销天数 |
| 订单级 | 单订单 | 完整成本拆解（瀑布图） |
| 广告级 | Campaign × 时间 | 利润口径 ROAS、销售 ROAS、利润额 |
| 关键词级 | Keyword × 时间 | 关键词利润贡献、ACoS 利润口径 |

#### 6.5.2 漏点检测（AI 主动）

预设规则 + AI 异常归因：

| 漏点类型 | 检测规则 | AI 输出 |
|---|---|---|
| 退货率突涨 | 单 SKU 7 日退货率 > 历史均值 + 2σ | 归因（变体？季节？质量？）+ 修复建议 |
| 长期仓储 | SKU 入仓 > 270 天 | 算账：清库 vs 留着哪个划算 |
| 广告亏损 | 利润 ROAS < 1 | 暂停 / 降出价建议 |
| 滞销资金占用 | SKU 7 日销量 = 0 且库存 > 30 天 | 真实成本估算（仓储 + 资金占用 + 机会成本） |
| 退款异常 | 单 SKU 退款率 > 类目均值 1.5x | 排查 Listing / 质量问题 |
| 价格倒挂 | 售价 < 完全成本（含分摊） | 立即预警，不卖比卖好 |

## 7. 模块 M3：生命周期广告优化

### 7.1 价值主张

> **AI 不只优化广告，它先判断这个 SKU 现在在生命周期哪一阶段，再用对应阶段该用的策略。把"AI 当真正的操盘手"，而不是"无脑批量调整工具"。**

### 7.2 用户故事

- **US-M3-1** 作为卖家，我要 AI 自动识别我每个 SKU 在哪个生命周期阶段。
- **US-M3-2** 作为卖家，我要不同阶段用不同策略，而不是同一套 ACOS 目标套所有 SKU。
- **US-M3-3** 作为卖家，我要在阶段切换时收到提醒（"该转手动了"、"该加预算了"）。
- **US-M3-4** 作为卖家，我要每天收到一份当日操作清单，能一键批量执行。
- **US-M3-5** 作为卖家，我要 AI 给的每个建议都附带"为什么"（推理链）。
- **US-M3-6** 作为卖家，我要按"利润 ROAS"而不是"销售 ROAS"来评估广告。
- **US-M3-7** 作为卖家，我要在广告效果可疑时（突增/突降）AI 主动调查归因。

### 7.3 生命周期识别

| 阶段 | 自动识别信号 | 默认策略目标 | KPI 容忍 |
|---|---|---|---|
| **新品期** | 上架 < 90 天 OR 评论 < 30 | 起量 + 拿评论 | ACOS 容忍 60-100% |
| **成长期** | 销量曲线连续 4 周上升、评论 30-100 | 扩规模 + 抢排名 | TACoS 看下降趋势 |
| **成熟期** | 销量平稳、BSR 稳定、评论 > 100 | 守排名 + 利润 | ACOS 严控 < 25% |
| **衰退期** | BSR 4 周连续下滑 + 库存高 OR 替代品出现 | 清库存 | 利润优先 |

识别方式：
- 规则引擎打初始标签
- LLM 复核（输入销量/BSR/评论数曲线、产品上架时长、类目竞争密度）
- 人工可手动覆盖

### 9.2a 跨模块共享数据（关键设计）

> 多个模块可能引用同一份数据。明确"主-从"关系防止重复存储 / 不一致。

#### 9.2a.1 关键词库（M1 ↔ M3 共享）

```
keyword_library 表（顶层共享）
  来源：
    - M3 搜索词报告自动入库（高曝光词、高转化词）
    - M1 用户标"主关键词 / 否定词"
    - 用户手动维护（关键词库管理页）
  
  消费：
    - M1：评分 D1 关键词覆盖度 + Listing 改写参考
    - M3：手动 Campaign 关键词扩展、否词候选
    - M4：核心词排名追踪的对象
```

**单一来源** = `keyword_library` 表（在数据底座，不属于任何单一模块）。

#### 9.2a.2 竞品数据（M1 / M2 / M4C 共享）

```
竞品快照（competitor_snapshots）= 全局共享
  - 同一竞品 ASIN 全平台只采集一次（成本优化 10-100 倍）
  - 每个租户在 competitor_products 中"关注列表"指向共享快照

用户特定数据（每租户独立）：
  - competitor_changes（事件流，按租户的关注列表过滤）
  - 用户私有标注 / 备注
```

**好处：**
- 数据成本降 10-100 倍（10000 个租户都关注同一头部竞品时）
- 数据更新一次更新所有租户视图

**注意：**
- 第三方数据合规（Keepa / SellerSprite ToS 是否允许 SaaS 转售）需在合同层面解决
- 用户上传的私有数据不进入共享池

| 简单分类 / 标签 | Claude Haiku 4.5 | GPT-4 Mini |
| Review 聚类 | Claude Sonnet 4.6 + Embedding | — |
| 多模态（图片评估） | Claude Opus 4.7 (vision) | GPT-4V |
| Embedding | text-embedding-3-large | bge-large |

### 10.3 Prompt 体系

#### 10.3.1 Prompt 模板结构

每个决策有 4 段：

```
[ROLE] 你是一个亚马逊广告专家，10 年经验，专注于成长期 SKU 的广告优化。

[CONTEXT]
- SKU 信息: {sku_meta}
- 当前生命周期: 成长期
- 近 30 天广告数据: {ads_metrics}
- 利润数据: {profit_data}
- 类目均值参考: {category_benchmarks}

[INSTRUCTION]
基于以上数据，给出今天的广告操作建议清单（5-10 条）。
每条必须包含：
1. 操作类型（加预算/调出价/否词/暂停/...）
2. 具体目标（哪个 Campaign / Keyword）
3. 推理依据（为什么这样改）
4. 预期影响（量化数字）
5. 优先级（高/中/低）

[OUTPUT FORMAT]
严格 JSON：{...schema...}
```

#### 10.3.2 Prompt 库管理

- 版本控制：每个 Prompt 有 v1/v2/...，可 A/B 测试
- 类目特化：电子配件 / 家居 / 母婴 等使用差异化 prompt
- Few-shot：积累优质案例库，按相似性检索注入

### 10.4 可解释性设计

每个 AI 决策必须包含：

```json
{
  "decision_id": "uuid",
  "recommendation": "...",
  "confidence": 0.85,
  "reasoning": {
    "summary": "一句话理由",
    "evidence": [
      {"type": "data", "source": "ads_metrics", "value": {...}},
      {"type": "rule", "rule_id": "growth_stage_bid_up", "matched": true},
      {"type": "benchmark", "vs_category": "+15%"}
    ],
    "considered_alternatives": [
      {"option": "...", "rejected_because": "..."}
    ]
  },
  "expected_impact": {"metric": "ROAS", "change": "+12%", "horizon_days": 14}
}
```

UI 上"AI 解释"按钮一点开就能看完整推理链。

### 10.6 审计与回溯

每个 AI 决策保留：
- 完整输入快照（数据 + Prompt）
- 完整输出（结构化 + 原始）
- 执行结果
- N 天后跟踪指标
- 用户操作日志（采纳/拒绝/微调）

支持：
- 按决策回溯调查
- 支持 90 天内自动执行的回滚
- 决策质量统计（采纳率 / 后效）

### 10.7 反馈学习闭环

- 用户对每个 AI 建议可标"有用 / 无用 / 拒绝原因"
- 拒绝原因汇总 → Prompt 迭代依据
- 采纳后 14/30 天效果 → 模型 fine-tune 数据集
- 高质量决策案例 → Few-shot 库

- 试用期开放"增长套餐"功能但不开放"操盘套餐"（防滥用）

### 12.3a Onboarding（首日体验，关键留存设计）

> ⚠️ 原文档说"30 分钟出第一份诊断报告"。技术上 SP-API 拉历史订单数据通常需要 1-3 小时，**30 分钟做不到完整诊断**。修订为分级体验：

| 时长 | 用户能看到 | 用什么数据 |
|---|---|---|
| **0-5 分钟（OAuth 后立刻）** | 店铺基础概览（店铺名、SKU 数、近 7 日 GMV 估算） | SP-API 实时元数据（轻量） |
| **5-30 分钟** | 单 SKU 快速诊断（M1 评分仅基于 Listing 文本，不含搜索词/Review） | SP-API Catalog Items（不依赖报告） |
| **30 分钟 - 4 小时** | M2 销售总览（不含完整利润，仅销售口径）+ M3 广告概览 | SP-API Reports（订单/广告基础） |
| **4-24 小时** | 完整 M2 利润（含成本估算）+ 漏点初步识别 | 报告完成 + AI 分析 |
| **24-72 小时** | 全模块完整诊断 + 第三方数据接入（如有套餐） | Keepa/SellerSprite 拉取完成 |

**UI 必须做的事：**
- 进度条 + "现在能看什么 / 24 小时后能看什么"清晰说明
- 每个完成阶段推送通知（"M3 广告数据已就绪 [查看]"）
- 7 天内"完整度 < 80%" 自动延长试用期 7 天（防数据未就绪导致的非真实流失）

### 14.5 跨 Session 续接机制

每次 session 开始：
1. 读 `MEMORY.md`（高密度索引，< 200 行）
2. 读 `PROJECT_STATUS.md`（当前进度 / 阻塞 / 下一步）
3. 读最近 git log（实际改了什么）
4. 跑 `docker-compose up` 启动开发环境
5. 跑 `make health-check` 验证所有服务正常

## 15. 数据合规与风险

### 15.1 SP-API / Ads API 合规

- ✅ 官方授权 OAuth 流程
- ✅ 不存储用户密码，只存 refresh_token（KMS 加密）
- ✅ 遵守 API 限流（动态退避）
- ✅ 记录所有操作日志（亚马逊审计要求）
- ⚠️ 用户可随时撤销授权 → 必须 24 小时内清除其历史数据

### 15.2 第三方数据源

| 数据源 | 合规要点 |
|---|---|
| Keepa | 商业 API，禁止再分发原始数据，可加工后用于决策 |
| SellerSprite | 同上 |
| Helium 10 | 同上 |

→ **不自行爬取亚马逊页面**，规避 ToS 风险。

### 15.3 用户数据保护

- 静态加密：DB / 备份 / 日志（敏感字段）
- 传输加密：全站 HTTPS + mTLS（服务间）
- 访问控制：RBAC + 审计日志
- 数据保留：用户注销 30 天后彻底删除
- 跨境合规：用户在 EU 选择 EU 部署区域（GDPR）

### 15.4 AI 决策法律风险

- **免责条款**：所有 AI 建议是"建议"，最终决定权在用户
- **全自动模式护栏**：金额上限、操作类型白名单、可一键回滚
- **审计可回溯**：任何执行都能查到决策时间、依据、推理
- **错误回滚**：发现 AI 决策错误，30 天内可一键回滚（Listing 改动除外）

### 15.5 系统风险

| 风险 | 缓解 |
|---|---|
| 第三方 API 中断 | 多源备份（Keepa + SellerSprite）+ 缓存兜底 |
| LLM API 中断 | 多模型路由 + 降级策略（规则兜底） |
| 数据库崩溃 | 主从 + 实时备份 + RPO < 5min |
| 单点故障 | 全部服务多副本 + 跨可用区 |
| 数据泄露 | 加密 + 审计 + 异常告警 |
| 恶意 AI 调用（abuse） | 速率限制 + 异常行为检测 |

### 17.3 优先级原则

如果时间紧张需要砍：
1. ✂️ 砍 M1（Listing 优化）— 最难做好，可后做
2. ✂️ 砍 M4C（竞品作战室）— 数据成本高
3. ✂️ 砍 全自动模式 — 风险高，先半自动稳一稳
4. ✂️ 砍 微信推送 — Phase 2 再加

不能砍：
- M2 利润 — 核心 PMF 验证点
- M3 广告 — 见效最快的销售证据
- 数据底座 — 没有它什么都做不了
